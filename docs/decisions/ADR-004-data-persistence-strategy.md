# ADR-004: 데이터 보존: 호스트 바인드 마운트 원칙

- **Status:** ✅ Accepted
- **Date:** 2026-04-06
- **Decider:** Gemini CLI & User

## Context (배경)

- 컨테이너는 본질적으로 휘발성(Ephemeral)이므로, 데이터의 영속성(Persistence) 확보 전략이 필수적임.
- 데이터 유실 방지뿐만 아니라, 장애 시 데이터에 즉각 접근하여 분석할 수 있는 투명성이 요구됨.

## Alternatives Considered (대안 분석)

1. **Docker Volumes (Named Volumes):** 도커가 관리하여 편리하나, 실제 물리적 저장 경로가 추상화되어(`/var/lib/docker/volumes/...`) 호스트 수준에서의 직접적인 모니터링 및 백업 접근성이 낮음.
2. **Cloud Storage (S3, EFS):** 네트워크 기반 지연 시간이 발생하며, 온프레미스/프라이빗 클라우드 환경에서는 추가적인 게이트웨이 설정이 필요함.

## Decision (결정)

- 데이터 가시성과 제어권 확보를 위해 **호스트 바인드 마운트(Bind Mount)**를 표준으로 함.
- `/data/[service_name]/`과 같이 명시적인 호스트 경로를 정의하여 컨테이너와 연결함.

## Consequences (결과)

- **이득:**
  - 컨테이너가 죽더라도 호스트 OS 수준에서 데이터를 즉시 확인 및 복구 가능.
  - 도커 명령어가 아닌 일반 백업 스크립트(rsync 등)로도 완벽한 데이터 자산화 가능.
- **비용:** 호스트 디렉토리의 UID/GID 권한 설정 불일치로 인한 쓰기 오류 발생 가능성이 있어, 서비스별 정밀한 권한 매핑이 필요함.
