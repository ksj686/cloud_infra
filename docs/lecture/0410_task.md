## 🔒 과제 1: HTTPS 사설 Docker Registry 구축 (hub.kosa.kr)

### 1️⃣ STEP 1. 환경 설정 및 hosts 등록

- **대상:** Registry 서버 (172.16.31.149) 및 모든 클라이언트

```bash
# 방화벽 설정

sudo ufw allow ssh
sudo ufw allow 443/tcp

# /etc/hosts 등록 (모든 서버 공통)

### 💡 /etc/hosts 영구 고정 방법 (cloud-init 환경)
- **원인**: cloud-init 도구가 부팅 시마다 `/etc/hosts`를 템플릿 기준으로 초기화함.
- **해결**: cloud-init 템플릿 파일에 직접 설정 추가.
- **절차**:
  1. 템플릿 파일 열기: `sudo vi /etc/cloud/templates/hosts.ubuntu.tmpl` (또는 `hosts.debian.tmpl`)
  2. 파일 맨 하단에 `172.16.31.149 hub.kosa.kr` 내용 추가 후 저장.
  3. 재부팅 시에도 해당 설정이 자동으로 포함되어 생성됨.
```

### 2️⃣ STEP 2. CA 및 SSL 인증서 생성 (`script.sh` 최적화)

`hub.kosa.kr` 도메인을 위한 자가 서명 CA와 인증서를 생성합니다.

```bash
mkdir -p ~/certs && cd ~/certs

# 1. Root CA 생성
openssl genrsa -out ca.key 2048
openssl req -x509 -new -nodes -key ca.key -days 10000 -out ca.crt \
  -subj "/C=KR/ST=Seoul/O=KOSA/CN=KOSA-CA"

# 2. Registry 서버용 개인키 및 CSR 생성
openssl genrsa -out domain.key 2048
openssl req -new -key domain.key -out domain.csr -subj "/CN=hub.kosa.kr"

# 3. SAN(Subject Alternative Name) 설정 파일 작성
cat <<EOF > extfile.cnf
subjectAltName = DNS:hub.kosa.kr, IP:172.16.31.149
EOF

# 4. CA로 인증서 서명
openssl x509 -req -in domain.csr -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out domain.crt -days 10000 -extfile extfile.cnf
```

### 3️⃣ STEP 3. 사용자 인증 및 Nginx 설정

계정명은 `kosa12` (예시), 비밀번호는 `kosa1004`로 생성합니다.

```bash
# apache2-utils 설치 및 계정 생성
sudo apt install apache2-utils -y
htpasswd -Bc htpasswd kosa12  # 비밀번호: kosa1004 입력

# Nginx 설정 파일 작성 (./certs/nginx.conf)
cat <<EOF > nginx.conf
upstream docker-registry {
  server registry:5000;
}

server {
  listen 443 ssl;
  server_name hub.kosa.kr;

  ssl_certificate /etc/nginx/conf.d/domain.crt;
  ssl_certificate_key /etc/nginx/conf.d/domain.key;

  client_max_body_size 0;
  chunked_transfer_encoding on;

  location /v2/ {
    auth_basic "Registry Realm";
    auth_basic_user_file /etc/nginx/conf.d/htpasswd;
    add_header 'Docker-Distribution-Api-Version' 'registry/2.0' always;

    proxy_pass http://docker-registry;
    proxy_set_header Host \$http_host;
    proxy_set_header X-Real-IP \$remote_addr;
    proxy_set_header X-Forwarded-For \$proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto \$scheme;
    proxy_read_timeout 900;
  }
}
EOF

### 💡 Nginx 설정 상세 설명
*   **`upstream docker-registry`**: 실제 Registry 컨테이너(`registry:5000`)를 백엔드로 정의합니다.
*   **`listen 443 ssl`**: HTTPS 통신을 위해 443 포트를 열고 SSL을 적용합니다.
*   **`client_max_body_size 0`**: 대용량 이미지 파일 Push를 위해 업로드 크기 제한을 해제합니다. (필수 설정)
*   **`auth_basic`**: `htpasswd` 파일을 사용하여 사용자 인증(ID/PW)을 수행합니다.
*   **`proxy_pass`**: 들어온 요청을 `upstream`으로 전달하며, `X-Forwarded-*` 헤더를 통해 클라이언트의 원본 정보를 유지합니다.
*   **`proxy_read_timeout 900`**: 대용량 파일 전송 중 연결이 끊기지 않도록 타임아웃을 15분으로 늘립니다.

```


## 🏗️ 과제 2: 내부 서비스 플랫폼 구축 (Nginx LB + API)

### 1️⃣ STEP 1. Backend Dockerfile 작성

요구된 `HEALTHCHECK`와 `STOPSIGNAL`을 정확히 반영합니다.

```dockerfile
FROM python:3.11-slim

# HEALTHCHECK를 위한 curl 설치 추가
RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY main.py .

# 환경변수 기본값
ENV APP_MODE=development
ENV APP_VERSION=unknown

# 과제 요구사항: 3s 후 시작, 30s 주기, 3회 재시도
HEALTHCHECK --interval=30s --timeout=5s --start-period=3s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# 과제 요구사항: SIGTERM 처리
STOPSIGNAL SIGTERM

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 2️⃣ STEP 2. 이미지 빌드 및 Push

```bash
# 이미지 빌드
docker build -t hub.kosa.kr/api-server:1.0 .

# 사설 저장소로 Push
docker push hub.kosa.kr/api-server:1.0
```

### 3️⃣ STEP 3. 서비스 배포 (3개 Backend + Load Balancer)

```bash
# 1. 네트워크 생성
docker network create kosa-net

# 2. Backend 컨테이너 3대 실행
for i in {1..3}; do
  docker run -d --name api-$i \
    --hostname api-$i \
    --network kosa-net \
    -e APP_MODE=production \
    -e APP_VERSION=1.0.0 \
    hub.kosa.kr/api-server:1.2
done

# 3. Load Balancer 설정 (api_lb.conf)
cat <<EOF > api_lb.conf
upstream backend_servers {
    # max_fails: 실패 횟수 제한, fail_timeout: 장애로 간주하는 시간
    server api-1:8000 max_fails=1 fail_timeout=10s;
    server api-2:8000 max_fails=1 fail_timeout=10s;
    server api-3:8000 max_fails=1 fail_timeout=10s;
}

server {
    listen 80;
    server_name api.kosa.kr;

    location / {
        proxy_pass http://backend_servers;
        proxy_set_header Host \$host;
        proxy_set_header X-Real-IP \$remote_addr;

        # 빠른 장애 대응을 위한 타임아웃 설정
        proxy_connect_timeout 2s; # 서버 접속 대기 시간 (2초)
        proxy_read_timeout 5s;    # 서버 응답 대기 시간 (5초)
        proxy_next_upstream error timeout http_500 http_502 http_503 http_504;
    }
}
EOF

### 💡 로드밸런서 최적화 설명
- **`max_fails/fail_timeout`**: 서버 장애 발생 시 즉시 감지하여 목록에서 일시적으로 제외함.
- **`proxy_connect_timeout`**: 서버가 죽어 있을 때 응답을 기다리는 시간을 대폭 줄여(60s -> 2s) 사용자 체감 로딩을 방지함.
- **`proxy_next_upstream`**: 현재 서버가 에러를 반환하거나 타임아웃이 나면 즉시 다음 서버로 요청을 재시도함.

# 4. LB 실행
docker run -d --name nginx_lb \
  --network kosa-net \
  -p 80:80 \
  -v $(pwd)/api_lb.conf:/etc/nginx/conf.d/default.conf \
  nginx:1.27

# 💡 설정 변경 시 핫 로드 (재시작 불필요)
# 호스트의 api_lb.conf를 수정한 후 아래 명령어를 실행하면 즉시 반영됩니다.
docker exec nginx_lb nginx -s reload
```

## 🔒 보안 강화: Docker 자격 증명 관리 (Credential Helper)

### 1️⃣ 현상 및 원인

- **경고 메시지**: `Your credentials are stored unencrypted in '/home/kosa/.docker/config.json'`
- **원인**: Docker는 기본적으로 로그인 정보를 Base64(평문 수준)로 저장하여 보안에 취약함.

### 2️⃣ 해결 방법: Pass(Credential Helper) 설정

OS의 암호 저장소를 사용하여 자격 증명을 안전하게 암호화

```bash
# 1. 필수 패키지 설치
# (Ubuntu/Debian에서는 golang-docker-credential-helpers 패키지에 포함되어 있음)
sudo apt update && sudo apt install pass golang-docker-credential-helpers -y

# 2. GPG 키 생성 (암호화용)
# 실행 후 이름, 이메일 입력 (실습에서는 Passphrase/비밀번호 비워둬도 무방. 현업에서는 설정 추천)
gpg --generate-key

# 3. 생성된 GPG 키 ID 확인
# pub 라인의 '/' 뒤에 있는 8자리 또는 16자리 문자열 확인 (예: ABC12345)
gpg --list-keys

# 4. Pass 초기화
# 위에서 확인한 ID 또는 등록한 이메일 입력
pass init "사용자이메일@example.com"

# 5. Docker 설정 반영
# ~/.docker/config.json 파일에 "credsStore": "pass" 추가
# (jq가 설치되어 있다면 아래 명령어로 한 번에 가능)
python3 -c "import json, os; path=os.path.expanduser('~/.docker/config.json'); d=json.load(open(path)); d['credsStore']='pass'; json.dump(d, open(path, 'w'), indent=2)"

# 6. 재로그인 (이제 경고가 사라짐)
docker login hub.kosa.kr
```

### 3️⃣ [운영 가이드라인] 환경별 보안 관리 표준

- **개발자 업무용 PC (Local)**
  - 보안 수준: **상(High)**
  - 권장 방식: **GPG 비밀번호(Passphrase) 설정** + `gpg-agent` 활용
  - 목적: 기기 분실 및 비인가자의 인증 정보 탈취 방지
- **운영/배포 서버 (Production/CI-CD)**
  - 보안 수준: **중(Medium)**
  - 권장 방식: **비밀번호 생략(Empty Passphrase)** + **권한 제어**
  - 보안 보완: 전용 Robot 계정(Read-Only) 사용 및 `.docker` 폴더 권한 엄격 제한(`chmod 700`)
- **클라우드/전문 도구 환경 (Enterprise)**
  - 보안 수준: **최상(Highest)**
  - 권장 방식: **Token 기반 로그인** 및 **IAM Role 활용**
  - 특징: 암호 저장 없이 클라우드 서비스(AWS IAM 등)나 외부 보안 솔루션(Vault)과 연동

### 4️⃣ 개발자 환경 편의 설정 (gpg-agent 자동화)

- **목적**: 세션 동안 암호를 메모리에 유지하여 반복 입력 번거로움 해소
- **설정 절차**
  1.  설정 파일 생성: `mkdir -p ~/.gnupg`
  2.  캐시 시간 설정: 초단위 ex)28800 은 8시간 유지

```bash
cat <<EOF > ~/.gnupg/gpg-agent.conf
default-cache-ttl 10
max-cache-ttl 10
EOF
```

3.  환경 변수 등록: `echo 'export GPG_TTY=$(tty)' >> ~/.bashrc`
4.  설정 적용: `source ~/.bashrc`
5.  에이전트 설정 새로고침: `gpg-connect-agent reloadagent /bye`

### 💡 [중요] gpg-agent 캐시 동작 원리 및 테스트 팁

- **Docker 로그인 vs GPG 캐시 차이**
  - `docker logout`: 자격 증명 저장소(`pass`)에서 정보를 **완전히 삭제**함. 따라서 재로그인 시에는 캐시와 상관없이 무조건 ID/PW를 새로 입력해야 함.
  - `gpg-agent`: 저장소에 있는 정보를 읽기 위한 **'마스터 열쇠(GPG Passphrase)'**를 메모리에 기억함.
- **올바른 캐시 테스트 방법**
  - `login/logout` 대신 `docker pull` 또는 `docker push` 명령어로 테스트할 것.
  - 1차 실행 시 GPG 암호 입력 -> 설정한 TTL 시간 이내 재실행 시 암호 생략 확인.
- **TTL(10초)이 지났는데도 암호를 묻지 않는 경우**
  - **원인 1 (Empty Passphrase)**: GPG 키 생성 시 비밀번호를 비워두었다면, 열쇠가 항상 열려 있는 상태이므로 암호를 묻지 않음. (실습 환경에서는 정상)
  - **원인 2 (OS Keyring)**: 데스크탑 환경(Ubuntu Desktop 등) 사용 시, OS 자체 키링(GNOME Keyring 등)이 GPG 암호를 별도로 가로채서 관리할 수 있음.
  - **확인**: 실제 현업 환경에서 보안을 확인하려면 반드시 GPG 키에 비밀번호를 설정하고 테스트할 것.

### 6️⃣ 완전한 로그아웃 및 인증 정보 제거 (Clean-up)

- **주의 사항**: 인자 없는 `docker logout`은 기본 Docker Hub만 로그아웃되므로 반드시 URL 명시 권장
