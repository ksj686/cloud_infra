# [Phase 3] Persistence: 데이터 및 스토리지 가용성

데이터 손실 방지 및 고가용성 저장소 구축을 위한 기술 실행 절차 정리

---

## 1. 가용 스토리지 구성 (RAID/LVM/ZFS)

하드웨어 결함에 대비한 물리 및 논리 볼륨 계층의 안정성 확보

- **RAID 어레이 구축 및 관리:**
  - RAID 1(미러링) 또는 RAID 5 구성을 통해 물리 디스크 장애 시에도 무중단 데이터 가용성 유지
- **사설 객체 저장소 (MinIO):**
  - AWS S3 호환 API를 로컬에 구축하여 클라우드 네이티브한 파일 관리 체계 확보
  - 향후 Phase 7(Hybrid Cloud) 이관을 위한 표준화된 데이터 연동 인터페이스 활용
- **LVM 볼륨 관리 고도화:**
  - LVM(Logical Volume Manager)을 활용한 가변적 파티션 설계 및 논리 볼륨(LV) 생성
  - 서비스 중단 없는 온라인 볼륨 확장(Resize) 및 축소(Reduce) 프로세스 정립
- **ZFS 무결성 검증:** 데이터 자가 복구(Self-healing) 및 카피-온-라이트(CoW) 스냅샷 기능을 위한 ZFS 미러링 도입 검토

## 2. 분산 스토리지 연동 (Ceph)

엔터프라이즈급 데이터 삼중화 및 공유 스토리지 인프라 구현

- **Ceph 클러스터 활성화:** Proxmox 하이퍼바이저 노드 간 Ceph 엔진 활성화 및 전용 복제 네트워크 구성
- **데이터 복제 정책 (Replication):** 네트워크 기반 삼중화 저장을 통한 스토리지 노드 단위의 결함 허용(Fault Tolerance) 능력 확보
- **공유 스토리지 풀(Pool) 관리:** 가상 머신(VM) 및 컨테이너가 즉각적으로 사용할 수 있는 탄력적인 RBD 스토리지 제공

## 3. 컨테이너 데이터 보존 전략

컨테이너의 일시적 생명주기와 분리된 영속성 데이터 관리 체계 수립

- **네임드 볼륨 (Named Volume) 활용:**
  - 데이터베이스(DB) 데이터 등 영구 보존이 필수적인 자산의 도커 통합 관리 및 격리 보호
  - **실행 예시:** `docker run -v db_storage:/var/lib/mysql mysql:latest`
- **바인드 마운트 (Bind Mount) 운용:**
  - `nginx.conf`, `prometheus.yml` 등 호스트 기반의 실시간 설정 동기화가 필요한 파일 연동
  - 호스트 물리 경로에 데이터를 직접 귀속시켜 백업 편의성 및 운영 가시성 확보

## 4. 엔터프라이즈 백업 및 복구 체계 (PBS)

재해 복구(DR) 전략 이행을 위한 중앙 집중식 이미지 레벨 백업 환경 구축

- **Proxmox Backup Server (PBS) 구축:**
  - 전용 백업 하드웨어 설치 및 클러스터 통합 관리 (IP 예시: `192.168.100.20:8007`)
  - 청크(Chunk) 단위 중복 제거 및 압축 기술을 통한 백업 스토리지 효율성 극대화
- **백업 스케줄링 및 무결성 검증:** 중요 서비스 데이터의 일 단위 자동 증분 백업 및 복구 테스트 정례화
- **재해 복구(DR) 전략 핵심 지표:**
  - **RPO (복구 시점 목표):** 주요 데이터의 최대 24시간 이내 유실 범위 제한 (일일 백업 기준)
  - **RTO (복구 시간 목표):** 스냅샷 및 증분 복원 기술을 활용한 수 시간 이내 서비스 정상화 지향
