# Policy: 백업 및 복구 정책 (Backup & Recovery)

## 📌 개요
- 데이터 가용성 보장 및 시스템 장애 발생 시 비즈니스 연속성(BCP) 확보함.
- RPO(복구 시점 목표) 1일, RTO(복구 시간 목표) 1시간 이내 준수 지향함.

## 🗂️ 1. 백업 대상 (Backup Scope)
- **운영체제 설정:** `/etc` 디렉토리(네트워크, SSH, 크론탭 등).
- **데이터베이스:** 주기적 SQL 덤프 파일 생성함.
- **서비스 데이터:** `/var/www` 또는 앱 소스 코드 디렉토리.
- **사용자 홈 디렉토리:** 중요 사용자 개인 데이터 및 설정(`/home`).

## 📅 2. 백업 주기 및 보존 (Schedule & Retention)
- **증분 백업(Incremental):** 매일 00:00 (월~토) 수행함.
- **전체 백업(Full):** 매주 일요일 00:00 수행함.
- **보관 주기(Retention Policy):**
    - **로컬 스토리지:** 최근 7일분 보관함.
    - **원격/클라우드 보관:** 최근 30일분(Full Backup 위주) 아카이브함.
    - **연간 보관:** 분기별 1회(Quarterly) 장기 보관함.

## 🛠️ 3. 백업 수행 및 저장 전략 (이원화)

### **Case 1. 자체 구축 DB (Self-Managed: MySQL, PostgreSQL 등 직접 설치 시)**
- **백업 방식:** `mysqldump` 또는 `pg_dump` 등 DB 전용 도구를 활용한 논리 백업 수행함.
- **자동화:** 크론탭(Cron)을 이용해 매일 새벽 자동 덤프 파일(.sql) 생성 및 압축함.
- **저장:** 생성된 덤프 파일을 외부 스토리지(NAS, S3 등)로 즉시 전송함.
- **관리:** 로컬 서버의 덤프 파일은 디스크 용량 관리를 위해 3일 경과 시 자동 삭제함.

### **Case 2. 클라우드 관리형 DB (Managed Service: AWS RDS, Azure SQL 등 사용 시)**
- **백업 방식:** 클라우드 제공사의 **자동 스냅샷(Automated Snapshots)** 기능을 활성화하여 관리함.
- **복구 지점:** **PITR(Point-in-Time Recovery)** 기능을 활성화하여 장애 발생 직전 시점(수 분 이내)으로의 정밀 복구 환경 확보함.
- **가용성:** **Multi-AZ(다중 가용 영역)** 설정을 통해 하드웨어 장애 시 자동 페일오버(Failover) 체계 구축함.
- **오프사이트 백업:** 클라우드 계정 탈취 등 최악의 시나리오 대비, 월 1회 SQL Export를 수행하여 타 계정 또는 로컬 스토리지에 별도 보관함.

## ✅ 4. 복구 검증 (Verification)
- **무결성 확인:** 백업 완료 후 파일 사이즈 및 체크섬(MD5/SHA256) 대조함.
- **복구 테스트:** 분기별 1회 임의의 백업본을 테스트 서버에 복원하여 서비스 정상 가동 여부 검증함.
- **훈련:** 장애 상황 시뮬레이션을 통해 복구 절차 숙달함.
